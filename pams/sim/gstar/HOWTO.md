# How to Define Particle Decays in `gstar_part.g`

## Overview

`gstar_part.g` is the STAR experiment's particle database for the GEANT3-based
simulation framework.  It defines the physical properties and decay modes of all
particles that the simulation is expected to propagate or decay, and it assigns
each one a GEANT3 particle ID (GID).

The file is written in **Mortran**, a FORTRAN-77 preprocessor language used by
the CERN GEANT3 libraries.  Lines beginning with `*` are Fortran-style comments
and are ignored at run time.  Inline comments begin with `!`.  Multi-line
particle definitions are terminated with a comma on each continuation line.

A machine-readable summary of all particle definitions is generated by
`parse_gstar_particles.py` (in this directory) and saved to
`GSTAR_PARTICLE_TABLE.md` at the repository root.


---

## The `PARTICLE` Statement

Every particle is introduced with the `PARTICLE` keyword followed by a name
and a set of key=value parameters, one or more per line, separated by commas.
The block ends at the next keyword that is not a continuation.

```fortran
  Particle <name> ["optional description"] code=<GID> ,
                  pdg=<pdg_id>  mass=<GeV>  charge=<e> ,
                  tlife=<s>     trktyp=<type>           ,
                  bratio={<f1>, <f2>, ...}              ,
                  mode={<m1>, <m2>, ...}
```

### Parameters

| Parameter | Type   | Description |
|-----------|--------|-------------|
| `code`    | int    | GEANT3 particle ID (GID). Must be unique. |
| `pdg`     | int    | PDG Monte Carlo ID. Use `UNDEFINED` (=0) when no official PDG code exists. |
| `mass`    | real   | Rest mass in GeV/c². |
| `charge`  | real   | Electric charge in units of the elementary charge *e*. |
| `tlife`   | real   | Mean lifetime in seconds. Use `STABLE` (=1.0×10¹⁵ s) for stable particles. |
| `trktyp`  | int    | Tracking type; governs which GEANT3 propagator subroutines are called (see below). |
| `bratio`  | array  | Branching ratios, one per decay mode, enclosed in `{ }` and separated by commas. Values need not sum to exactly 1 — GEANT3 normalises them. Omit for stable particles. |
| `mode`    | array  | Decay mode codes, one per entry in `bratio` (see decay mode encoding below). |

An optional quoted string immediately after the name is treated as a
human-readable description by GEANT3 and is ignored by the simulation logic:

```fortran
  PARTICLE LamXi2430 "Lambda0 Xi0 bound state" code=60002 ...
```

A trailing `_` (underscore) on a continuation line is an alternative
line-continuation character in Mortran:

```fortran
  PARTICLE XiZeroBar1530 _
                      code=40008 ...
```

### Named Constants Available in the File

```fortran
Integer UNDEFINED / 0       /  ! Use for unknown PDG IDs
Real    STABLE    / 1.0E+15 /  ! Use for stable particles
Real*8  kHBAR = 6.58211889E-25 ! GeV·s  (use for tlife = kHBAR/Width)
```

## Tracking Types (`trktyp`)

The `trktyp` parameter tells GEANT3 which set of propagator subroutines to call.
The following named integer constants are defined at the top of the module:

| Constant   | Value | Particle class | Typical use |
|------------|-------|----------------|-------------|
| `kGtGAMA`  | 1     | Photon         | γ |
| `kGtELEC`  | 2     | Electron/positron | e±, conversion electrons |
| `kGtNEUT`  | 3     | Neutral hadron | neutral pions, neutral kaons, neutral baryons |
| `kGtHADR`  | 4     | Charged hadron | charged pions, kaons, protons, charged baryons |
| `kGtMUON`  | 5     | Muon           | μ± |
| `kGtNINO`  | 6     | Geantino       | ghost/test particles that leave no ionisation |
| `kGtCKOV`  | 7     | Cherenkov photon | optical photons |
| `kGtHION`  | 8     | Heavy ion      | nuclei, anti-nuclei, hyper-nuclei |

> **Rule of thumb**: use `kGtNEUT` (3) for charge=0 hadrons, `kGtHADR` (4) for
> charge≠0 hadrons, and `kGtHION` (8) for anything with |charge| ≥ 2 or with
> A > 1 nuclear content.

Note that value **9** appears for the magnetic monopole (`Monopole`, GID=800)
and is not a standard GEANT3 tracking type — it is handled through custom code.

## Decay Mode Encoding

The `mode` array specifies the daughters of each decay channel.  Each entry is
a concatenation of two-digit GEANT3 IDs with no separator.  The order must
match the corresponding entry in `bratio`.

**Two-body decay** — two daughters, four digits total:

```fortran
mode = { 1409, }   ! proton(14) + pi-(09)
```

**Three-body decay** — three daughters, six digits total:

```fortran
mode = { 040803, }  ! pi0(04) + pi-(08) ... wait, actual:
                    ! pi-(04=K0L, 08=pi+, 03=nu_e) -> K0L pi+ nu_e
```

**Multiple channels** — one entry per branching ratio:

```fortran
bratio = { 0.69, 0.31 }
mode   = { 3708, 3507 }   ! 69% D*+ → D0(37)+π+(08);  31% D*+ → D+(35)+π0(07)
```

> **Tip**: daughter IDs are the GEANT3 built-in IDs of the standard particles
> (see the table in the next section), or the custom GIDs of any helper
> particles defined elsewhere in the file.

## GEANT3 Particle ID Numbering Conventions

GEANT3 uses a single integer GID to identify every particle.  IDs 1–50 are
reserved for the standard built-in particles listed above.  All other IDs in
this file follow the conventions below.

### Sub-ranges in Use

| GID range | Convention | Examples |
|-----------|-----------|---------|
| 1–50      | Standard GEANT3 built-ins (read-only) | proton=14, pion=8/9, Λ=18 |
| 51–54     | Alias IDs for special nuclei | 52=H3_Lambda, 53=antiDeuteron, 54=antiHelium3 |
| 92–99     | Helper/constrained-decay intermediaries, counted **backwards from 99** | 99=_phi_to_KK_, 98=_lam_to_p_piminus_, 97–92 see below |
| 100–799   | Short non-standard IDs for resonances and specials | 149=Dalitz, 150=ω, 151=φ, 701–704=Σ(1385) |
| 800       | Magnetic monopole | 800=Monopole |
| 995–996   | Lambda(1520) variants | StLambda1520, StLambdaBar1520 |
| 707       | K⁰_S (short-lifetime variant) | StK0s |
| 10000+N   | Redefinitions of built-in particle N with specific forced decay | 10011=K⁺→μ⁺νμ, 10018=Λ→p+π⁻ |
| 11000+N … 15000+N | Additional forced-decay variants of the same built-in | 11011=K⁺→K⁰π⁺, 12011=K⁺→π⁺π⁺π⁻, … |
| 40000+    | Non-standard baryons not in GEANT3 built-ins | 40001=Ω̄⁺, 40003=Ξ⁻, 40005=Ξ⁰ |
| 50000+    | Anti-nuclei and nuclear excited states | 50045=antiDeuteron, 50060=Be8, 51045=Deuteron |
| 60001–60009 | Dibaryons, di-mesons, exotic bound states | 60001=H dibaryon, 60002=LamXi2430 |
| 60008–60009 | Pentaquarks | PQ1730 |
| 60100+    | Pseudoparticles (loosely bound compound states) | 60100–60103 d-hyperon states |
| 60801     | Strangelets | H0strangelet |
| 61000+    | Hyper-triton / hyper-helium with specific decay modes | 61053=hyperTriton_he3_pi_minus |
| 62000+    | Hyper-triton with deuteron+proton+π decay mode | 62053 |
| 63000+    | Hyper-triton quasi-DL mode (user-buffer decay) | 63053 |

### The `10000+N` Redefinition Pattern

When you need a specific forced-decay branching for a particle that GEANT3
already knows about, define it with `code = 10000 + <built-in GID>`.  You can
define multiple variants of the same particle using `10000+N`, `11000+N`,
`12000+N`, etc.

```fortran
! Lambda constrained to decay 100% to p + pi-
PARTICLE Lambda  code=10018  trktyp=kGtNEUT  mass=1.115683 ,
                 charge=0    tlife=0.26320E-9               ,
                 bratio={1.000,}  mode={1409,}
```

### The Helper Particle Pattern

Helper particles (GIDs 92–99) are *not* real physical particles.  They
represent a specific particle **constrained to a single forced decay channel**,
and they are used exclusively as daughters in the `mode` arrays of higher-level
particles.  This lets multi-step decay chains be represented without requiring
GEANT3 to know about the intermediate steps.

Current helper assignments (counting backwards from 99):

| GID | Name | Decay |
|-----|------|-------|
| 99  | `_phi_to_KK_`            | φ → K⁺ K⁻ |
| 98  | `_lam_to_p_piminus_`     | Λ → p π⁻ |
| 97  | `_lam_to_pb_piplus_`     | Λ̄ → p̄ π⁺ |
| 96  | `KAON_PLUS`              | K⁺ → π⁺π⁺π⁻ (forced) |
| 95  | `K0short`                | K⁰_S (forced) |
| 94  | `_sigma0_to_lam_gamma_`  | Σ⁰ → Λ(→pπ⁻) γ |
| 93  | `_xi_minus_to_lam_pi_`   | Ξ⁻ → Λ(→pπ⁻) π⁻ |
| 92  | `_xi_zero_to_lam_pi0_`   | Ξ⁰ → Λ(→pπ⁻) π⁰ |

When you need a new helper, assign the next available ID counting backwards
(91, 90, …).  Add the helper *before* the parent particle that references it,
so GEANT3 can resolve the ID at registration time.

## PDG Ion ID Helper Functions

For nuclei and hyper-nuclei, the file provides two Fortran helper functions
that compute the PDG Monte Carlo number from proton number Z, mass number A,
and (for hypernuclei) strangeness content L:

```fortran
PDG_ION_ID(Z, A)    = 1000000000 + 10000*Z + 10*A
PDG_HYP_ID(Z, A, L) = PDG_ION_ID(Z,A) + 10000000*L
```

Usage in a particle definition:

```fortran
PARTICLE antiDeuteron  code=50045  trktyp=kGtHION
                       mass=1.876  charge=-1  tlife=STABLE ,
                       pdg=-PDG_ION_ID(1,2)
```

## User-Buffer Decay (Direct `GSPART` Call)

For decays that involve more than three daughters, or when a particle's decay
products include a custom GID that Mortran cannot express in the compact `mode`
syntax, call `GSPART` directly using the `uw` (user words) buffer:

```fortran
Particle hypertriton_quasi_DL  code=63053  mass=2.99130 ,
                               charge=+1   tlife=2.6320e-10 ,
                               pdg=UNDEFINED  trktyp=kGtHADR

    uw = { 0, 11018, 51045 }   ! mode: FastLambda(11018) + Deuteron(51045)
    Call GSPART( %code, %title, %trktyp, %mass, %charge, %tlife, uw, nw )
```

The first element of `uw` is a reserved flag (set to 0 for a standard two-body
phase-space decay encoded in the buffer).  The remaining elements are GEANT3
particle IDs.  `nw` is predefined as 3 (the length of the user words array).

When using this form, **omit** `bratio` and `mode` from the `PARTICLE`
statement — the decay is registered entirely through the direct call.

## Step-by-Step: Adding a New Particle

1. **Choose a GID.**  Pick from the appropriate sub-range (see the table above).
   For a new exotic hadron or pseudoparticle with no standard GEANT3 equivalent,
   use 60100+ or the next available ID in the relevant cluster.

2. **Check for a helper requirement.**  If the decay chain passes through an
   intermediate state that must itself be constrained to a single channel,
   define that helper first (GID ≤ 99, counting backwards from 92).

3. **Ensure mass conservation.**  The parent mass must strictly exceed the sum
   of daughter masses so that decay products carry non-zero momentum.

4. **Choose `trktyp`.**  Use `kGtNEUT` for neutral hadrons, `kGtHADR` for
   charged hadrons, `kGtHION` for heavy ions.

5. **Encode the decay mode.**  Concatenate two-digit GIDs of the daughters
   left-to-right.  Two-body → four digits; three-body → six digits.

6. **Insert the block.**  Place it in the appropriate section of the file,
   after any helper particles it references.  Add a `*` comment above the block
   explaining the decay chain, the sum of daughter masses, and a contact
   reference if applicable.

7. **Update the parser and table.**  Add a matching rule to the `GROUPS` list
   in `parse_gstar_particles.py` (if the particle belongs to a new family) and
   re-run the script to regenerate `GSTAR_PARTICLE_TABLE.md`.

### Minimal Example

```fortran
* New scalar meson S* (hypothetical) --> pi+ pi-
* sum of daughters: 0.140 + 0.140 = 0.280 GeV
  Particle S_star    code=60200  trktyp=kGtNEUT ,
                     mass=0.400  charge=0  tlife=1.0E-20 ,
                     pdg=UNDEFINED ,
                     bratio={1.0,} ,
                     mode={0809,}
```

## The Decay Chain for `d-Hyperon` Pseudoparticles

These pseudoparticles illustrate the full multi-step helper chain pattern.

```
d_Lambda  (60100) → d(45)  + _lam_to_p_piminus_(98)
                              └→ p(14) + π⁻(09)

d_Sigma0  (60101) → d(45)  + _sigma0_to_lam_gamma_(94)
                              └→ _lam_to_p_piminus_(98) + γ(01)
                                  └→ p(14) + π⁻(09)

d_Xi_minus(60102) → d(45)  + _xi_minus_to_lam_pi_(93)
                              └→ _lam_to_p_piminus_(98) + π⁻(09)
                                  └→ p(14) + π⁻(09)

d_Xi_zero (60103) → d(45)  + _xi_zero_to_lam_pi0_(92)
                              └→ _lam_to_p_piminus_(98) + π⁰(07)
                                  └→ p(14) + π⁻(09)
```

Parent masses were chosen to exceed the sum of daughter masses so that all
final-state products are kinematically accessible.

| Pseudoparticle | Parent mass (GeV) | Daughter mass sum (GeV) | Margin (MeV) |
|----------------|:-----------------:|:-----------------------:|:------------:|
| d_Lambda       | 2.998             | 1.876 + 1.115683 = 2.992 | 6 |
| d_Sigma0       | 3.074             | 1.876 + 1.193    = 3.069 | 5 |
| d_Xi_minus     | 3.204             | 1.876 + 1.32171  = 3.198 | 6 |
| d_Xi_zero      | 3.204             | 1.876 + 1.31486  = 3.191 | 13 |

## Maintenance

- **Parser**: `pams/sim/gstar/parse_gstar_particles.py`  
  Re-run after any change to `gstar_part.g` to regenerate `GSTAR_PARTICLE_TABLE.md`
  and verify three-way ID consistency.

- **Verification**: The script checks that:
  1. Every GID found by the parser appears in the raw text of `gstar_part.g`.
  2. Every GID in the generated Markdown table appears in `gstar_part.g`.
  3. Every GID in `gstar_part.g` appears in the Markdown table.

- **Commented-out placeholders**: GIDs 60053 and 60054 appear only in
  `*`-comment lines as placeholder stubs for a future full-multiplicity
  hypertriton definition.  They are intentionally excluded from the parsed
  table; the working definitions live at 61053/61054, 62053/62054, and
  63053/63054.
